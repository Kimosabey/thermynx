# ADR — Per-task LLM right-sizing

**Status:** Accepted · 2026-06-01
**Decided by:** Harshan (project owner) + operations findings during the post-`5f8b38e` verification run
**Last updated:** 2026-06-01

Sibling docs:

---

## Context

Through `4d2b0c9` we added per-task model env vars (`OLLAMA_MODEL_TEXT`, `_TOOL`,
`_SQL`, `_PLANNER`, `_AUDITOR`) but left them empty — every AI call still used
the single 14B model. After `5f8b38e` we filled in the defaults to right-size.

Verification (~2026-06-01):
- Agent path latency: ~10s → **~8s warm** (target met)
- Self-critique pass: now uses 3B, much faster
- **NL-Query: 1-in-3 transient validator-reject** rate on a complex aggregate
  query when generated by `llama3.1:8b` instead of `qwen2.5:14b`
- Total VRAM hot set went from 9 GB (one model) → **~16 GB (three models)**

The NL-Query flake raised the question: should we switch SQL to the bigger
purpose-built coder model?

---

## Decision

**Keep `llama3.1:8b` for NL-to-SQL.** Do not switch to `qwen2.5-coder:32b`
or revert to `qwen2.5:14b`.

Final per-task mapping (locked in `backend/app/config.py` commit `5f8b38e`):

| Task slot | Setting | Model | Param count | VRAM (KV q8_0) |
|---|---|---|---|---|
| `OLLAMA_MODEL_TEXT` | `""` (→ default) | `qwen2.5:14b` | 14.8B | ~9 GB |
| `OLLAMA_MODEL_TOOL` | `"llama3.1:8b"` | `llama3.1:8b` | 8.0B | ~5 GB |
| `OLLAMA_MODEL_SQL` | `"llama3.1:8b"` | `llama3.1:8b` | 8.0B | (shared slot with TOOL) |
| `OLLAMA_MODEL_PLANNER` | `"llama3.1:8b"` | `llama3.1:8b` | 8.0B | (shared slot) |
| `OLLAMA_AUDITOR_MODEL` | `"llama3.2:latest"` | `llama3.2:latest` | 3.2B | ~2 GB |
| `OLLAMA_VISION_MODEL` | `"llama3.2-vision"` | `llama3.2-vision` | 10.7B | ~8 GB (swaps) |
| `OLLAMA_DEFAULT_MODEL` | `"qwen2.5:14b"` | fallback | — | — |

Operationally on the Ollama host: `OLLAMA_MAX_LOADED_MODELS=3` keeps the
three primary models hot. Vision + embedder load on demand and rotate the
LRU slot.

## Alternatives considered

### A1 — Switch `OLLAMA_MODEL_SQL` to `qwen2.5-coder:32b` (rejected)

**Why we looked at it:** Purpose-built for code/SQL, would likely reduce the
1-in-3 validator-reject rate to near zero.

**Why rejected:**
1. **VRAM budget breaks.** Adding 19.9 GB (Q4_K_M) on top of the current 16 GB
   hot set = ~36 GB. Doesn't fit on a 24 GB GPU, definitely doesn't fit on
   16 GB. Returns us to model thrash.
2. **Slower than the current 8B.** 32B params → ~20 tok/s vs 8B's ~60-80 tok/s.
   Net latency would be ~3s per SQL gen vs current ~1-1.5s warm.
3. **Optimizes the wrong surface.** NL-Query is the lowest-stakes AI surface;
   power-user feature, not main demo path. Trading 8B speed-and-VRAM for 32B
   correctness on a feature most operators rarely touch is wrong priority.

### A2 — Revert `OLLAMA_MODEL_SQL` to `""` (use default `qwen2.5:14b`) (rejected)

**Why we looked at it:** Simpler topology — only 2 models hot
(`qwen2.5:14b` + `llama3.2:latest`). Eliminates the 8B SQL flake by reusing
the 14B already loaded for narration.

**Why rejected:**
1. **Loses the agent speedup.** `OLLAMA_MODEL_TOOL` would still be `llama3.1:8b`,
   so we'd still have 3 models hot (TOOL=8B, TEXT=14B, AUDITOR=3B). No VRAM
   saving.
2. **14B isn't dramatically better at SQL than 8B.** Both produce structurally
   valid SELECTs ~70%+ of the time. The 30% delta is on complex aggregates,
   and on those neither model is reliable — both need operator iteration.
3. **Adds latency without proportionate benefit.** 14B SQL gen at ~30 tok/s vs
   8B at ~60-80 tok/s. NL-Query slows from ~1.5s to ~2.5s warm for a barely-
   noticeable correctness improvement.

### A3 — Add a retry chain (try 8B → if validator rejects → retry with 14B) (rejected)

**Why we looked at it:** Best of both — speed on common case, correctness
fallback on flakes.

**Why rejected:**
1. **Doubles worst-case latency** — failed cases pay 8B time + 14B time + 2
   round-trips to Ollama.
2. **Hides the signal** that the operator's question is genuinely ambiguous.
   A clear refusal teaches the operator to rephrase; a silent retry papers over
   communication gaps.
3. **Complexity tax.** Adds a code path that needs its own eval cases, error
   handling, observability. Wrong investment for a low-stakes surface.

---

## Consequences

### Good
- **Agent latency cut ~20%** (warm 8s vs ~10s pre-tune) on the main demo path
- **Self-critique cost cut ~5×** (3B vs 14B) — runs on every analyzer answer
- **Eval suite ~13% faster** (87s vs ~100s) from the same model-sizing effect
- **VRAM budget fits in 16 GB** with KV q8_0 — works on common GPU tiers
- **Operators can override any one slot via env var** — no code changes
  required to bump SQL back to 14B or up to coder:32b later

### Bad / accepted
- **NL-Query has a ~30% validator-reject rate on complex aggregate queries.**
  Operator sees a clean 422 with a clear reason and rephrases. We accept this
  as the validator working as designed.
- **First call after a long idle pays ~3-5s cold-load** on whichever model
  rotated out. Mitigated by `OLLAMA_KEEP_ALIVE=30m` + the pre-warm step in
  the tuning script.
- **Operationally requires 3 hot model slots** on the Ollama host
  (`OLLAMA_MAX_LOADED_MODELS=3`). On 12 GB GPUs the operator must drop to 2
  and accept SQL-side cold-loads.

---

## Triggers to revisit this decision

We will re-open this ADR if any of these become true:

1. **Audit log shows >20% NL-Query validator-reject rate** sustained over
   1+ week of real operator usage. Right now we observed 1-in-3 on one
   deliberately complex query — not a representative sample. If real usage
   shows this is a common operator complaint, the trade-off changes.

2. **A second facility deploys** with a different equipment catalog and
   SQL queries become more complex by necessity (e.g. multi-tenant joins,
   role-based filters). The 8B might no longer suffice.

3. **GPU upgrade puts us at 40 GB+ VRAM** (RTX 6000 Ada or A100 class).
   At that point coder:32b can stay hot alongside qwen2.5:14b + llama3.1:8b
   and the speed-vs-correctness trade-off flips.

4. **A new base model lands** (qwen3, llama4, etc.) with significantly
   better SQL at the 7-8B parameter tier. Re-evaluate at that point.

5. **Operator feedback loop ships** (planned in [EVALUATION_PLAN.md](./EVALUATION_PLAN.md))
   and accumulated 👎 votes on NL-Query show the model is the bottleneck,
   not phrasing.

## Verification snapshot (2026-06-01)

For provenance:

```
=== Latency benchmark (3 runs each — first = cold-load) ===
  NL-Query simple                  cold=  4.9s  warm_avg=  2.9s
  NL-Query medium                  cold=  3.5s  warm_avg=  3.3s  (1/3 validator reject)
  Agent investigator               cold= 10.5s  warm_avg=  8.4s
  Analyzer + critique              cold= 27.0s  warm_avg= 27.9s

=== Eval regression check ===
  19 passed in 87.48s
```

Once the operator deploys the updated `ollama_restart_tuned.bat` (commit
`0eb5702`) with `MAX_LOADED_MODELS=3` + pre-warm:
- NL-Query warm should drop from 2.9s → **~1.5s**
- Agent cold should drop from 10.5s → **~5-7s**
- Self-critique should be a stable **0.5-1s**

These targets are projected, not yet measured against the new host config.

## How to override (operator)

If a future operator wants to swap any task back without re-opening this ADR,
set the env var in `.env` or the process environment. No code change required:

```bash
# Best-quality SQL (slower, needs ~24 GB VRAM headroom):
OLLAMA_MODEL_SQL=qwen2.5-coder:32b

# Or revert SQL to the 14B default:
OLLAMA_MODEL_SQL=

# Or test a smaller model for tool selection:
OLLAMA_MODEL_TOOL=llama3.2:latest

# Or use llama3.1:8b for narration too (uniform-8B setup):
OLLAMA_MODEL_TEXT=llama3.1:8b
```

The override doesn't invalidate this ADR — it documents what we'd run by
default if no one was experimenting.
